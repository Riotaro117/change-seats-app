# 🪑 はじめに

教員が直感的に教室の座席配置を作成、管理するためのWebアプリケーションです。

https://change-seats-app.vercel.app

ご利用いただくことでのトラブル等は自己責任でお願いします。

## 💡 コンセプト
教員である友人が「席替えのアプリはたくさんあるけど、きちんと配慮すべきことを配慮してランダムに席替えできるアプリがない」という要望をもとに、要件定義・設計・実装をしました。

## ✨ 主な機能

- **ランダムシャッフル機能**: ワンクリックで配慮事項を踏まえて、生徒の座席をランダムに再配置
- **座席の設定**: 実際の教室を模したグリッドUIで座席を配置、座席数、列数の指定、使用しない座席の設定
- **生徒管理**: アプリ内で生徒情報を一元管理（名前、性別、前列配慮、相性の配慮）、追加、編集、削除が可能
- **データ連携**: `xlsx` (SheetJS) を用いた名簿データのシームレスな入出力機能に対応
- **履歴**: 過去の座席をグリッドレイアウトの中から選んで復元
- **保存**: 現在の座席配置を名前をつけて保存
- **印刷**: 完成した座席表を名前のみ抽出してプリンタで印刷・PDF出力
- **クリックによる座席の移動**: 任意の座席をクリックすることで入れ替え
- **認証**: 仮登録からユーザー本登録までをルートベースで堅牢に制御 (Supabase Auth)
- **モダンで直感的なUI**: Lucide ReactアイコンとTailwind CSSによるレスポンシブなデザイン

## 🚀 環境構築と起動

リポジトリを適当なディレクトリにクローン後、以下の手順でローカル環境を構築してください。

### 1. 依存パッケージのインストール

Node.js環境 (推奨 18.x以上) が整っていることを確認し、依存パッケージをインストールします。

```bash
npm install
```

### 2. 環境変数の設定

プロジェクトのルートディレクトリに `.env.local` ファイルを作成し、ご自身の Supabase プロジェクトの資格情報を設定してください。これらの値は Supabase ダッシュボードの「Project Settings > API」から取得できます。

```env
VITE_SUPABASE_URL=your_supabase_url
VITE_SUPABASE_ANON_KEY=your_supabase_anon_key
```

### 3. 開発サーバーの起動

```bash
npm run dev
```

ターミナルに表示されたローカルURL（通常は `http://localhost:5173` ）にブラウザでアクセスすると、アプリケーションが起動します。

## 📁 ディレクトリ構成の概要

本アプリケーションは、関心の分離（Separation of Concerns）を明確にし、長期的な保守性と拡張性を担保するため、以下のようなディレクトリ構造を採用しています。

```text
src/
├── components/          # 汎用的に再利用可能なUIコンポーネント群（ボタン、入力フォーム、ダイアログなど）
│   ├── layout/          # ヘッダーやナビゲーションなど、画面のひな形となるレイアウト要素
│   └── ui/              # アプリ全体で共通して使われるアトミックなUIパーツ
├── modules/             # ドメイン関心事・機能ごとの分割（Feature-Sliced Design的アプローチ）
│   ├── auth/            # 認証・ユーザー情報に関するコンポーネントやロジック
│   ├── layouts/         # 座席のグリッド表示や配置に関するドメイン機能
│   └── students/        # 生徒名簿の管理・Excelインポート/エクスポートなどのドメイン機能
├── page/                # ページコンポーネント（URLの各パスに対応するエントリーポイント）
│   ├── Home.tsx         # メインの座席管理画面（保護されたルート）
│   ├── Signin.tsx       # ログイン・新規登録画面（パブリックルート）
│   ├── UpdateUser.tsx   # アカウント本登録用のユーザー情報入力画面（仮ユーザールート）
│   └── ...
├── Routes/              # アクセス権限に基づくルーティングコンポーネント（Auth Guard）
│   ├── ProtectedRoute.tsx     # 本登録済みユーザーのみ許可する設定
│   ├── AnonymousOnlyRoute.tsx # 仮登録（ログイン済み・未本登録）ユーザーのみ許可する設定
│   └── PublicOnlyRoute.tsx    # 未ログインユーザーのみ許可する設定
├── contexts/            # React Context（Jotai と並行して使用する広域のテーマや認証情報の提供など）
├── lib/                 # アプリケーション外の技術スタックの設定（Supabase クライアントの初期化など）
├── utils/               # 汎用・純粋なユーティリティ関数（日時フォーマット、データの計算・変換処理など）
├── type.ts              # アプリケーション全体で使用される共通のTypeScript型定義
└── main.tsx / App.tsx   # アプリケーションのルート設定（DOMへのマウント、プロバイダの設定、ルーティング）
```

## 🔒 認証とルーティング

Supabase Auth と React Router を連携させ、ユーザーの登録・ログイン状態に応じたきめ細かいアクセス制御（ルートガード）を実装しています。未認可のアクセスをクライアントサイドで適切にハンドリングし、セキュリティとUX（ユーザー体験）の向上を図っています。

### アクセス権限に基づくルート種別

本アプリケーションには、大きく分けて3種類のルート（経路）が存在し、それぞれ適切なコンポーネント（HOC的に振る舞うラッパーコンポーネント）で保護されています。

1. **保護されたルート (`ProtectedRoute`)**
   - **対象**: `/` など
   - **条件**: サインイン済み **かつ** アプリケーションが要求するプロフィールの初期設定（本登録）が完了しているユーザーのみアクセス可能。
   - **挙動**: 未ログインの場合は `/signin` へ、アカウント情報が登録途中の場合は `/updateUser` へリダイレクトさせます。

2. **仮ユーザー用ルート (`AnonymousOnlyRoute`)**
   - **対象**: `/updateUser`
   - **条件**: サインインは完了しているが、まだ初期設定や必要な追加情報の登録を終えていないユーザー（仮登録状態）のみアクセス可能。
   - **挙動**: すでに本登録を完了しているユーザーがアクセスした場合は `/` へリダイレクトし、サインインすらしていないユーザーは `/signin` へリダイレクトさせます。強制的に初期設定のフローへと誘導するためのルートです。

3. **パブリックルート (`PublicOnlyRoute`)**
   - **対象**: `/signin`, `/registered`
   - **条件**: まだシステムにログインしていない、未認証のユーザー向け。
   - **挙動**: すでにログイン済みで本登録まで完了しているユーザーがアクセスした場合は `/` へ、仮登録状態のユーザーは `/updateUser` へリダイレクトさせます。ログイン完了後に再びログイン画面を踏ませないための制御です。

### 認証フロー
1. ユーザーが `/signin` でメールアドレス・パスワード、またはOAuth等を用いてSupabase Authでログインします。
2. ログイン成功後、Supabase セッションが取得されます。
3. このとき、データベース上にユーザープロフィールの詳細なレコードが存在するかどうかをチェックします。
4. なければ `/updateUser` へ遷移し、必要な情報（氏名など）を入力させます。
5. 全ての認証・情報入力が完了して初めて `/` にアクセス可能となり、メインの座席管理機能を利用できるようになります。

## 📄 実装予定の機能
- **指定席実装**（1名のみ席を固定してあとはランダムに席替えを実行）
- **生徒ラベリング機能実装**（名前の横に５色ぐらいラベルを表示し、色を指定するとマーキングされる）
- **過去の座席との比較機能実装**（履歴をもとに、過去に隣同士の座席に座ったことのある生徒を知らせる）
- **黒板と座席が反転して見える実装**（座席表を反転させて、所見を書きやすくする）
